# CLAUDE.md — AI Assistant Guidance for hermes-senpi-trader

This file provides guidance to Claude, Gemini, and other AI coding assistants when working with this repository.

## Project Overview

This is a **Railway deployment template** for [NousResearch Hermes Agent](https://github.com/NousResearch/hermes-agent) with [Senpi MCP](https://senpi.ai) integration for autonomous crypto trading on Hyperliquid.

**What this repo contains:**
- Bootstrap and lifecycle scripts for the container
- Canonical YAML config templates (providers, MCP, risk policy, skill manifests)
- A Python config renderer (`scripts/render-config.py`) that converts registry YAML + env vars into Hermes-compatible generated config
- Default workspace content (`defaults/workspace/`)
- Backup, restore, and health check utilities

**What this repo does NOT contain:**
- Runtime application code (Hermes itself is pulled as a Docker image)
- Web UI
- Trading logic (that belongs in skills and workspace prompts)

## Architecture Mental Model

```
Env vars (secrets)          Canonical YAML (config)
      │                            │
      └──────────────┬─────────────┘
                     ▼
            render-config.py
                     │
          ┌──────────┴──────────┐
          │                     │
  .hermes/analysis/      .hermes/execution/
  config.generated.yaml  config.generated.yaml
          │                     │
    hermes (analysis)    hermes (execution)
```

**Critical invariants:**
1. `config.generated.yaml` is never edited by hand — regenerated by `render-config.py` on every boot
2. Secrets (`OPENROUTER_API_KEY`, `SENPI_AUTH_TOKEN`, etc.) live only in env vars, never in YAML files
3. Execution profile requires `risk_policy.yaml` to exist and pass validation
4. Senpi MCP (`profiles: [execution]`) never loads in the analysis profile

## Development Commands

```bash
# Local smoke test (Docker required)
docker build --build-arg HERMES_VERSION=v0.6.0 -t hermes-senpi-trader .

docker run --rm \
  -e ACTIVE_PROFILE=analysis \
  -e DATA_DIR=/data \
  -e OPENROUTER_API_KEY=your-key \
  -v $(pwd)/.tmpdata:/data \
  hermes-senpi-trader

# Dry-run config generation (no Docker needed)
python3 scripts/render-config.py \
  --profile analysis \
  --data-dir ./test-data \
  --dry-run

# Run healthcheck against a local /data
DATA_DIR=./test-data bash scripts/healthcheck.sh

# Test risk policy validation
DATA_DIR=./test-data bash bootstrap/validate-risk.sh
```

## Config System Rules

### `provider_registry.yaml`
- `api_key_env` must reference an env var name — never the value itself
- `enabled: false` providers are skipped by the renderer
- `supports_tools: true` required for MCP-capable providers

### `model_aliases.yaml`
- All agent prompts reference aliases (`trade-safe`, `analysis-deep`, etc.), never raw model IDs
- The `profiles:` block at the bottom sets the `default_alias` and `fallback_alias` per profile
- When adding a new alias, the referenced `provider` must exist in `provider_registry.yaml`

### `mcp_registry.yaml`
- `profiles: [execution]` — only loads in execution profile (Senpi MCP is locked here)
- `profiles: [analysis, execution]` — loads in both
- `enabled: false` servers are never generated into `mcp.generated.yaml`
- `env:` values use `${ENV_VAR_NAME}` format — resolved at render time

### `risk_policy.yaml`
- `symbols.default_action` **must** be `deny` — validation rejects anything else
- `symbols.allowlist` must have at least one entry
- `mode: copilot` → all actions require human approval (recommended start)
- `execution.allow_open_new_position: false` is the safe default

### `skills/manifests/*.yaml`
- `source.pinned_commit` is mandatory and must not be a placeholder — `install-skill.sh` enforces this
- `policy.enabled: false` → skill is installed but not active; operator enables manually
- `policy.dry_run_override: true` → skill cannot bypass global dry_run setting

## Bootstrap Boot Sequence

```
prestart.sh
  [1] Volume mount check        → DATA_DIR must exist
  [2] First-run init            → if version.json missing, run init-volume.sh
  [3] Migration check           → run-migrations.sh compares applied.log to migrations/
  [4] Risk policy validation    → only in execution profile; validate-risk.sh
  [5] Config generation         → render-config.py produces generated YAML
  [6] Hermes runtime start      → exec hermes --config config.generated.yaml
```

Each step is idempotent — safe to run on every container restart.

## Adding New Providers

1. Add env vars to `.env.example` (with empty value)
2. Add provider block to `config/providers/provider_registry.example.yaml`
3. Add one or more aliases to `config/providers/model_aliases.example.yaml`
4. Document env vars in README Quick Start table
5. `render-config.py` reads `api_key_env` and resolves from `os.environ` automatically

No code changes needed — it's purely declarative.

---

## Adding New MCP Servers

1. Add a server block to `config/mcp_registry.example.yaml`
2. Set `profiles:` appropriately (`[analysis]`, `[execution]`, or both)
3. Set `risk_level:` — used for human review/audit, not enforced programmatically
4. Add any required env vars to `.env.example`
5. `render-config.py` filters by `ACTIVE_PROFILE` and generates `mcp.generated.yaml` automatically

---

## Adding New Skills

1. Create a new manifest YAML in `config/skills/`
2. Set `source.pinned_commit` to the exact git SHA
3. Set `policy.enabled: false` initially
4. Document in README Skills table
5. Copy to `/data/skills/manifests/` and run `install-skill.sh` to install

## Migration System

Migration files are numbered bash scripts in `migrations/`:

```
migrations/
├── 0001_initial_schema.sh   # first-run schema setup
├── 0002_add_health_logs.sh  # future example
└── runner.sh                # NOT a migration — called by run-migrations.sh
```

- Files are applied in sort order
- Applied files are tracked in `/data/.runtime/migrations/applied.log`
- Each migration must be **idempotent** — safe to re-run
- Once a migration is applied, do not modify it

## Common Gotchas

1. **`config.generated.yaml` is stale** → Happens when you edit a canonical YAML but don't restart. Always restart the container after config edits.

2. **Execution profile won't start** → `validate-risk.sh` exited with error. Check: (a) `/data/risk/risk_policy.yaml` exists, (b) `symbols.default_action: deny`, (c) `symbols.allowlist` non-empty, (d) YAML is valid.

3. **Senpi MCP not connecting** → Check `SENPI_AUTH_TOKEN` env var is set and not expired. Verify `mcp_registry.yaml` has `enabled: true` and `profiles: [execution]`.

4. **Skill install fails with "branch-only install rejected"** → Set `source.pinned_commit` in the skill manifest to a real commit SHA from the skill repo.

5. **render-config.py warning: "no valid provider alias resolved"** → At least one provider must have `enabled: true` and a non-empty API key env var. Check Railway Variables.

6. **Container healthcheck fails** → Run `bash scripts/healthcheck.sh` inside the container to see which check is failing.

## Security Guidelines

See [SECURITY.md](SECURITY.md) for the full security model.

**When working on this codebase, never:**
- Print or log API key values (env var names are fine)
- Write secrets to any file under `/data`
- Set `symbols.default_action: allow` in examples
- Use `latest` as an image tag or `pinned_commit`

**Always:**
- Use `api_key_env: ENV_VAR_NAME` pattern (not the value)
- Use `set -euo pipefail` in all bash scripts
- Mark generated files with a `# GENERATED` header comment
- Keep `policy.enabled: false` as the default for all new skills

## Railway Deployment Notes

- Template must mount a volume at `/data` — without it, nothing persists
- Set `ACTIVE_PROFILE=analysis` initially — execution is opt-in
- `HERMES_VERSION` is pinned via Docker build arg in `railway.toml`
- Public networking must be enabled if Telegram webhook is used
- For upgrade: update `HERMES_VERSION` in `railway.toml`, take a backup first, then redeploy
